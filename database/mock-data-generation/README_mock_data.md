## generate_119.py

### Tables generated

#### `request_comments`

| Column | Type | Notes |
|---|---|---|
| `comment_id` | bigint (PK) | Auto-incrementing |
| `req_id` | varchar(255) | FK → request table |
| `commenter_id` | varchar(255) | FK → users table (Cognito UUID format) |
| `comment_desc` | text | Category-specific realistic comment text |
| `created_at` | timestamp | Always after the parent request's created_at |
| `last_updated_at` | timestamp | Always ≥ created_at |
| `isdeleted` | boolean | ~5% TRUE (realistic soft-delete rate) |

#### `volunteer_rating`

| Column | Type | Notes |
|---|---|---|
| `volunteer_rating_id` | integer (PK) | Auto-incrementing |
| `user_id` | varchar(255) | FK → users table |
| `request_id` | varchar(255) | FK → request table |
| `rating` | enum (1–5) | Skewed positive: 45% five-star, 33% four-star |
| `feedback` | text | Sentiment matches rating value |
| `last_update_date` | timestamp | Within 14 days of request creation |

### How to run

```bash
# Navigate to this folder
cd database/mock-data-generation

# Default: 100 rows per table (recommended for current use)
python generate_119.py

# Custom row count
python generate_119.py --rows 500

# Custom random seed (different data each run)
python generate_119.py --rows 100 --seed 99

# Scale test (see Data Quality note below before using for ML)
python generate_119.py --rows 40000
```

### Using external CSVs for full FK integrity

Once `users.csv` and `request.csv` are generated by the other team,
regenerate these tables to ensure all foreign keys are consistent
across the entire synthetic dataset:

```bash
python generate_119.py \
  --users-csv database/mock_db/users.csv \
  --requests-csv database/mock_db/request.csv
```

The script auto-detects column name variants in external CSVs and maps
raw category strings to the official Saayam category hierarchy.

### Known limitations and upgrade path

#### Text diversity at scale

**Planned upgrade — LLM-generated text:**

For ML-quality synthetic data at scale, comment_desc and feedback fields
should be generated via LLM API calls (GPT-4o, Gemini, or Claude) using
category-specific prompts. This produces high-diversity, semantically
varied text that trains better NLP classifiers and sentiment models.

Example approach (future implementation):

```python
# Planned: replace fixed templates with LLM generation
def generate_comment_llm(category: str, subcategory: str) -> str:
    prompt = f"""Write a realistic one-sentence volunteer comment for a 
    Saayam help request in the '{subcategory}' subcategory under '{category}'.
    Sound like a real person — not formal, not lorem ipsum."""
    return call_llm_api(prompt)
```

This upgrade is tracked as a follow-on improvement to this issue.
Current 100-row output is sufficient for all immediate development and testing needs.

#### Referential integrity with real data

Until users.csv and request.csv are finalized by the other team, this script
generates its own internal user and request ID pools. Run with `--users-csv`
and `--requests-csv` flags once those files are available to ensure full
cross-table FK consistency.
